WS : Composer / MVC / PHP

Objectifs du WS

Ce workshop vous donne certaines clés pour architecturer votre projet, gérer les dépendances tierces et garantir (dans une certaine mesure) la qualité du code. Plus concrètement, il vise à :

  1. Comprendre l’architecture MVC (Modèle-Vue-Contrôleur).
  2. Utiliser Composer pour la gestion des dépendances.
  3. Utiliser le moteur de template Twig.

Vous allez structurer votre première application MVC en développant MyTodo, un outil simple et efficace pour gérer vos tâches quotidiennes. MyTodo se compose de deux pages principales : une page d’accueil (home) pour créer et lister vos tâches en cours, et une page d’historique (history) qui archive les tâches terminées et permet de les rétablir (‘undo’) en un clic. Ce projet vous permettra de mettre en pratique les concepts clés du MVC (Modèle-Vue-Contrôleur), d’utiliser un moteur de template comme Twig pour dynamiser vos vues, et de gérer les interactions utilisateur avec une base de données légère.

1. Installation et configuration

⚠️ Pour avoir un environnement fonctionnel dans le cadre du projet My Todoo, vous devez réaliser au préalable les étapes suivantes :

  • Installer Composer (voir les ressources du prosit)
  • Créer une entrée DNS locale mytodoo.local -> 121.0.0.1 (Windows)(Linux)
  • Créer un répertoire qui contiendra le code du microprojet MyTodoo (par exemple :/var/www/my_todoo)
  • Créer un VirtualHost qui redirige mytodoo.local vers le répertoire du projet (voir le prosit sur Apache)
  • Copier le contenu du projet my_todoo dans le répertoire que vous venez de créer (par exemple :/var/www/my_todoo)

2.3 La vue

Pour générer la vue, il est tout à fait possible d’utiliser du PHP comme nous l’avons fait jusque-là. Seulement, le mélange entre le code HTML et PHP rend très souvent le code confus. Par ailleurs, beaucoup d’aspects visuels se répètent d’une page web à l’autre (comme la top bar, le footer, les menus latéraux…). Les moteurs de template répondent à ces problématiques et à bien d’autres en proposant une séparation claire entre la logique de présentation et la logique métier. Ils offrent une syntaxe simplifiée et plus lisible pour intégrer des variables et des structures de contrôle dans le HTML, ce qui rend les fichiers de vue plus compréhensibles. De plus, les moteurs de template facilitent la réutilisation des blocs de code grâce à des fonctionnalités comme l’héritage de templates, les inclusions, et les fonctions/filtres/macros, réduisant ainsi la duplication de code et améliorant la modularité de l’application. Ils intègrent également des mécanismes de sécurité pour échapper automatiquement les variables, prévenant ainsi les attaques par injection XSS. Enfin, ils permettent une meilleure collaboration entre les designers et les développeurs, chacun pouvant se concentrer sur son domaine d’expertise tout en utilisant un langage de template commun.

Consigne : Vous devez créer un template Twig pour notre application de gestion de TODO List. Le template doit afficher une liste de tâches à faire, permettre l’ajout de nouvelles tâches et marquer les tâches comme terminées. Le template devra être enregistré dans le répertoire template avec le nom todo.twig.html.

Les données : Les données envoyées au template sont sous la forme d’un tableau de tâches, où chaque tâche est représentée par un dictionnaire contenant un identifiant (id) et le contenu de la tâche (task). Par exemple :

[
    ["id" => 1, "task" => "Message 1"],
    ["id" => 2, "task" => "Message 2"],
    ...
]

Le code HTML de base est fourni ci-dessous. Vous devez intégrer les données dynamiques à l’aide de Twig.

<html>

    <head>
        <meta charset="utf-8">
        <title>Home</title>
        <meta name="viewport" content="width=device-width, initial-scale=1">
        <link rel="stylesheet" href="https://unpkg.com/mvp.css"> 
    </head>

	<body>
	    <header>
	        <nav>
	            <h1>✅ My Todoo </h1>
	            <ul>
	                <li><a href="/">Home</a></li>
	                <li><a href="/?uri=history">History</a></li>
	                <li><a href="/?uri=about">About</a></li>
	            </ul>
	        </nav>
	        
	        <form action="/?uri=add_task" method="post"> 
	            <h2>📝 Add a new task</h2>   
	            <input name="task" placeholder="Add a new task" style="width: 100%;"/>
	            <button type="submit">Add</button>
	        </form>
	        
	    </header>
	    <main>
	        <h2>My todo list</h2>
	        <!--Loop over tasks-->
	        <form action="/?uri=check_task" method="post">
	            <label>task1</label>
	            <input name="id" type="hidden" value="1">
	            <button type="submit">Done</button>
	        </form>    
	        <form action="/?uri=check_task" method="post">
	            <label>Things to do ...</label>
	            <input name="id" type="hidden" value="4">
	            <button type="submit">Done</button>
	        </form>    
	        <!--...-->    
	    </main>
	</body>
</html>

Indications sur la logique de présentation :

  1. Boucle sur les tâches : Utilisez une boucle Twig pour parcourir les tâches et afficher chaque tâche dans un formulaire.
  2. Affichage des tâches : Pour chaque tâche, affichez le contenu de la tâche dans un label et l’identifiant de la tâche dans un champ caché (input de type hidden).
  3. Formulaire de marquage : Chaque tâche doit avoir un bouton “Done” qui soumet un formulaire pour marquer la tâche comme terminée.

🚀 Consigne complémentaire : Cette consigne est à réaliser, si le workshop est intégralement terminé. Utilisez la directive extends de Twig pour créer une hiérarchie de templates sur plusieurs niveaux, vous permettant de factoriser du code HTML présent sur plusieurs vues.

2.4 Le contrôleur

Pour rappel, le contrôleur agit comme un intermédiaire entre le modèle (qui gère les données et la logique métier) et la vue (qui affiche les données à l’utilisateur). Dans le contexte de l’application MyToDoo, le contrôleur TaskController est responsable de gérer les interactions de l’utilisateur, telles que l’ajout de nouvelles tâches, le marquage des tâches comme terminées, et l’affichage des différentes pages de l’application. Il interagit avec le modèle TaskModel pour récupérer et manipuler les données des tâches, et utilise un moteur de template pour rendre les vues HTML.

Consigne : Complétez l’implémentation de la classe TaskController. Complétez en priorité les méthodes suivantes : welcomePage, addTask, checkTask.

🚀 Consigne complémentaire : Les méthodes : historyPage, uncheckTask, aboutPage doivent être réalisés dans un second temps, quand vous avez terminé l’intégralité du workshop. Les méthodes historyPage et aboutPage devront faire l’objet de la création des templates correspondants.

Indications :

  • Méthode welcomePage : Récupérez la liste des tâches à partir du modèle en utilisant une méthode appropriée (par exemple, getAllTasks). Passez cette liste de tâches au moteur de template pour rendre la vue todo.twig.html.

  • Méthode addTask : Vérifiez si le paramètre task est présent dans la requête POST. Si le paramètre n’est pas présent, redirigez l’utilisateur vers la page d’accueil. Récupérez la valeur du paramètre task. Appelez la méthode addTask du modèle avec la tâche comme paramètre. Redirigez l’utilisateur vers la page d’accueil.

  • Méthode checkTask : Vérifiez si le paramètre id est présent dans la requête POST. Si le paramètre n’est pas présent, redirigez l’utilisateur vers la page d’accueil. Récupérez la valeur du paramètre id. Appelez la méthode checkTask du modèle avec l’identifiant comme paramètre. Redirigez l’utilisateur vers la page d’accueil.

  • Méthode historyPage : Récupérez la liste des tâches à partir du modèle. Rendez la vue history.twig.html avec la liste des tâches.

  • Méthode uncheckTask : Suivez les mêmes étapes que pour la méthode checkTask, mais appelez la méthode uncheckTask du modèle à la place.

  • Méthode aboutPage : Rendez la vue about.twig.html.

2.5 Le routeur

Le routeur agit comme un point d’entrée principal de l’application, analysant les URL des requêtes entrantes et dirigeant ces requêtes vers les méthodes appropriées des contrôleurs. En d’autres termes, le routeur détermine quelle action doit être exécutée en fonction de l’URL demandée par l’utilisateur. Dans le contexte de notre application, le routeur est responsable de diriger les requêtes vers les méthodes spécifiques du contrôleur TaskController, telles que l’affichage de la page d’accueil, l’ajout de nouvelles tâches, ou l’affichage de l’historique des tâches.

Consigne : Complétez le code du routeur en appelant les méthodes appropriées du contrôleur TaskController en fonction de l’uri de la requête. Concentrez-vous en priorité sur les Uri : /, addTask. Une fois ces ajouts terminés, testez votre application.

🚀 Consigne complémentatire : Dans un second temps, si vous avez terminé le workshop, complétez les autres uri. Assurez-vous de gérer les cas où l’uri n’est pas reconnu en retournant une erreur 404.
