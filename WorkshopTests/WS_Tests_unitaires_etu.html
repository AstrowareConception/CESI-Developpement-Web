Premiers pas avec les tests unitaires et PHPunit

1. Testons l’application 📚Bookshop

Cette application est disponible dans une archive à part sur Moodle, téléchargez-la et placez-la dans votre environnement de développement PHP. N’oubliez pas d’installer les dépendances avec Composer.

1.1 Préambule sur les tests

Prenez connaissance du projet Bookshop en parcourant les fichiers qui le composent. Puis, répondez aux questions suivantes :

❓Quelles classes vont faire l’objet de tests dans ce projet ?
❓Où sont écrits les tests ?
❓D’après la documentation de PHPUnit, quelle est la convention pour définir le nom des classes de test ? De quelle classe, les classes de tests doivent-elles hériter ? Comment sont nommées les méthodes d’une classe de test ?
❓Comment vérifie-t-on dans un test unitaire (écrit avec PHPUnit) qu’une valeur obtenue correspond bien à la valeur attendue ?
❓Quelle commande permet d’exécuter les tests ?

1.2 Réalisation des tests unitaires de la classe Book

Consigne : Après avoir installé les dépendances du projet, complétez les méthodes testGetAuthor, testGetIsbn de la classe BookTest en vous inspirant de la méthode testGetTitle. Puis, exécutez vos tests et assurez-vous qu’ils sont tous validés. (N’hésitez pas à utiliser l’option --testdox)

❓ Au début de chaque test, il faut instancier un nouvel objet Book, comment pouvons-nous éviter cette répétition dans le code ?

1.3 Réalisation des tests unitaires de la classe Library

Consigne : Passons à présent aux tests de la classe Library. Sur le même modèle que la classe Book, écrivez les tests de la classe Library. N’oubliez pas de placer le code qui permet d’initialiser chacun de vos tests dans la méthode setUp.

2. Testons une application plus complexe : ✅MyTodo

⚠️ Nous réutilisons le code de notre application MyTodo, il est nécessaire que le workshop sur le MVC soit terminé et fonctionnel.

Les tests réalisés dans la partie précédente portent sur des classes assez simples qui dépendent peu, voire pas du tout, d’autres classes. Si nous revenons dans notre projet My Todoo, la situation est différente. Le contrôleur maintient des dépendances vers le moteur de template et le modèle. Le modèle dépend quant à lui de la base de données (avec l’attribut $connection). Si toutes nos classes maintiennent des dépendances, comment réaliser des tests unitaires ? (pour rappel, un test unitaire porte sur une classe et la réussite ou l’échec d’un test ne doit pas être affecté par une dépendance).

❓ En vous appuyant sur la documentation de PHPUnit, expliquez en quoi les mocks et les stubs permettent de répondre à ce problème. Qu’est-ce qui différencie un mock d’un stub ?

Consigne #1 : Dans la classe TaskModelTest (qui se trouve dans tests/TaskModelTest.php), complétez la méthode testAddTask.

Consigne #2 : Toujours dans la classe TaskModelTest, complétez la méthode testGetDoneTasks en vous inspirant de la méthode testGetToDoTasks. (⚠️ Il faudra avoir implémenté la méthode getDoneTasks de la classe TaskModel ).
